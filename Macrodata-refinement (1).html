What started as a Python script to organize chaotic Excel files became a lesson in installation design, user autonomy, and what it means to build something that works without you.

02
Research

Who I
designed for.

Small company, small audience. My users were my colleagues, which meant I could observe the problem daily and ask questions without scheduling a single meeting.

Primary User

The Logistics Lead

  • 40+. Excel expert, zero programming.
  • Knows the process better than anyone.
  • Relies entirely on manual methods.
  • High tech risk aversion — if something can go wrong, she assumes it will.

Needs

speed + control

Secondary User

The Art Director

  • Hates spreadsheets. Actively avoids them.
  • Similar technical profile, different relationship with the task.
  • Doesn't want to master the process, she wants it gone.
  • High initial distrust, but once convinced: needs to work independently.

Needs

autonomy + documentation

The Key Insight

“Make it do it by itself.” “…and I need to see what it's doing. And don't delete anything.”

Unanimous answer. Immediately followed by the real constraint. Full automation was the desire — supervised automation was the actual need.

03
The
Approach

First a sketch. Then proof.
Then the product.

Five decisions shaped the tool. Each one was a response to something observed, not assumed.
01

The Proof-of-Concept

Wrote a simple Python script that did one thing: reorder columns A and B. Nothing else. Proof that the problem was automatable. It worked: I got the space to develop a real version while keeping my other responsibilities.

02

Diagnosis

Asked users directly: what scares you? What confuses you? What would be ideal? The answers defined both the goal and its limits. They wanted full automation, but they also needed final say on every output. The program would have to propose, never impose.

03

Flow Design

Two database types required two logics. “Blue” databases (predictable structure): auto-process in seconds. “White” databases (variable structure): detect pattern, show preview, wait for user confirmation before executing. Users select the type manually, not because the program couldn’t detect it, but because at that stage, user trust was more reliable than program accuracy.

04

Retro Interface

The Windows-era terminal aesthetic was intentional, not a technical limitation. Users didn’t need a sophisticated-looking app. They needed one that felt simple, controllable, predictable. The visual message had to match the task: this is straightforward, you are in control.

05

Installation Redesign

First version required Python and Git, an invisible wall for non-technical users. Nearly killed the project before launch. Redesigned as a one-click executable: no visible dependencies, no terminal, one-page manual. Installation became part of the design, not an afterthought.

05 The
Learning.

“The hardest part wasn’t building the program. It was making it work on someone’s computer who didn’t ask for it. Installation is design.”

01

Architecture first,
code second.

I learned Python solving each bug as it appeared, without a clear logic structure underneath. If I did it again, I’d talk to someone who knows programming before writing the first line — not to delegate, but to build on a better foundation.

02

Installation
is design.

Making it installable was almost harder than making it work. The first version required Git management — something completely outside my users’ scope. That nearly killed the project. The printed manual, the one-click executable, the walkthrough session — all of that was design work, not afterthought.

03

Build a maintenance
channel from the start.

I no longer work at the company. The tool is still in daily use, but I have no control over feedback or future changes. If I could redo the project, I’d build a maintenance channel from the start — a way for the tool to evolve without depending on my presence.

Artefact

Manual de Procedimiento Operativo.

A printed booklet designed so that anyone in the office could install and run the program without calling me. Bureaucratic tone, deliberate structure, zero ambiguity. Part of the design process, not separate from it.
